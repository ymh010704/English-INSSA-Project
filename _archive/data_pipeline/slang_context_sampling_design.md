# Slang Context Sampling Design

## 1. 목적

이 단계의 목적은 **대표 sense를 최종 확정하는 것**이 아니다.  
현재 단계의 목적은 다음 하나로 한정한다.

> **후속 sense 분포 분석이 가능할 만큼 slang evidence를 충분히 확보하는 것**

즉, 지금 단계에서는 각 단어에 대해 문맥을 샘플링하고, 그중 실제 slang 용법으로 분류된 문맥 수를 일정 수준까지 확보하는 것이 핵심이다.

---

## 2. 핵심 변수

- `match_count (M)`  
  해당 표면형이 커뮤니티 코퍼스에서 **의미와 무관하게** 몇 번 등장했는지 나타내는 값

- `sampled_contexts (n)`  
  실제로 샘플링하여 확인한 문맥 수

- `slang_hits (K)`  
  샘플링한 문맥 중 **slang 용법으로 판정된 문맥 수**

- `floor ratio (p0)`  
  설계상 보수적으로 가정하는 최소 slang 비율  
  본 설계에서는 다음과 같이 둔다.

```text
p0 = 5%
```

---

## 3. 왜 floor ratio를 5%로 두는가

5%는 “실제로 쓰이는 slang의 진짜 최솟값이 반드시 5%다”라는 뜻이 아니다.  
이 값은 **초기 탐색에서 너무 빨리 drop하지 않기 위한 보수적 하한선**이다.

현재까지 확인한 실제 slang 예시들에서는 다음과 같은 경험적 비율이 관찰되었다.

| 단어 | slang 비율 |
|---|---:|
| cap | 8.0% |
| bomb | 16.8% |
| dope | 19.2% |
| shorty | 39.2% |
| geek | 45.6% |
| savage | 46.2% |
| peep | 50.8% |
| slay | 57.2% |
| snitch | 70.2% |
| drippy | 72.0% |
| red pill | 84.2% |
| sesh | 87.8% |
| chill | 88.2% |
| banger | 93.6% |
| yoink | 93.6% |
| crash out | 94.0% |
| bro | 98.4% |
| legit | 98.8% |

이 리스트에서 볼 수 있듯, 실제로 자주 쓰이는 slang은 대체로 5%보다 훨씬 높게 나타났다.  
따라서 5%는 **실제 positive의 평균값**이 아니라, **보수적인 floor**로 해석하는 것이 맞다.

정리하면:

- **5%**: 초기 drop을 너무 공격적으로 하지 않기 위한 보수적 floor
- **8%**: 실제 확인된 lower anchor (`cap`)
- **15~20% 이상**: 더 강한 slang zone (`bomb`, `dope` 이상)

---

## 4. 왜 초기 샘플 수를 60으로 고정하는가

초기 샘플 수가 너무 작으면, 실제로 slang 비율이 5%인 단어조차 우연히 slang이 하나도 안 나올 수 있다.

slang 비율이 5%일 때, 문맥 `n`개를 샘플링했는데 slang hit가 **0개** 나올 확률은 대략 다음과 같다.

```text
P(K = 0) ≈ (1 - 0.05)^n = 0.95^n
```

예시:

- `n = 20` → `0.95^20 ≈ 35.8%`
- `n = 50` → `0.95^50 ≈ 7.7%`
- `n = 60` → `0.95^60 ≈ 4.6%`

즉,

- 20개만 보면 진짜 5%짜리 단어도 3분의 1 넘게 0개가 뜰 수 있다.
- 60개를 보면 0개 확률이 약 4.6%로 내려간다.

따라서 초기 샘플을 20~30개로 두는 것은 너무 공격적이며,  
**초기 샘플 60개**는 최소 탐지용 기준으로 타당하다.

중요한 점은 이 확률이 **`match_count`가 아니라 `p`와 `n`에 의해 결정된다**는 것이다.  
즉, `match_count`가 10K든 1M이든, floor를 5%로 두고 60개를 보면 0-hit 확률은 동일하게 약 4.6%다.


---

## 5. `match_count`를 어떻게 써야 하는가

`match_count`는 slang 비율을 직접 말해주지 않는다.  
해당 표면형이 커뮤니티에서 **얼마나 자주 등장했는지**만 보여준다.

따라서 `match_count`로 바로 keep/drop을 결정하면 안 된다.

### 올바른 사용법

- **초기 샘플 수(`n0`)를 줄이는 데 사용하지 않는다**
- **추가 탐색의 최대 상한(`n_max`)을 정하는 데 사용한다**

이유는 다음과 같다.

- 초기 탐지의 신뢰성은 `p`와 `n`에 달려 있으므로, 모든 단어에 공통 최소 샘플 60개가 필요하다.
- 반면 `match_count`가 큰 단어는 문맥 풀이 크므로, 더 많은 evidence를 쌓을 가치가 있다.

즉,

> **초기 탐지는 공통 60개로 수행하고, `match_count`는 추가 탐색 상한을 조절하는 데만 사용한다.**

---

## 6. 왜 중단조건을 `K >= 15`로 두는가

### 6.1 8개는 너무 낮다

누적 slang hit가 8개면 “분포가 존재한다”는 정도는 볼 수 있지만, 후속 sense 비교를 위한 최소 분포로 보기에는 아직 약하다.

예를 들어 8개는 다음과 같은 분포만 봐도 해석이 쉽게 흔들린다.

- `5 : 3`
- `4 : 4`
- `4 : 3 : 1`
- `3 : 3 : 2`

문맥 1~2개만 추가되어도 top1/top2 해석이 쉽게 뒤집히므로, 8은 **진입선**으로는 가능해도 **기본 중단선**으로는 약하다.

### 6.2 20개는 더 안정적이지만 운영 비용이 커진다

20개는 분포 비교 측면에서 더 낫다.  
하지만 약한 실제 slang (`cap` 수준, 약 8%)까지 고려하면 필요한 샘플 수가 빠르게 커진다.

예상 샘플 수는 대략 다음과 같다.

```text
필요 샘플 수 ≈ 목표 hit 수 / slang 비율
```

예시:

#### 목표 `K = 15`
- `p = 8%` → 약 `188`개
- `p = 16.8%` → 약 `89`개
- `p = 19.2%` → 약 `78`개

#### 목표 `K = 20`
- `p = 8%` → 약 `250`개
- `p = 16.8%` → 약 `119`개
- `p = 19.2%` → 약 `104`개

즉,

- **15개**는 약한 실제 slang도 상위 bucket에서는 도달 가능한 수준
- **20개**는 더 안정적이지만 전체 운영 비용이 눈에 띄게 증가함

### 6.3 따라서 15개는 기본 중단조건으로 균형이 좋다

현재 설계 목표는 “대표 sense 확정”이 아니라, **후속 분포 분석이 가능한 최소 evidence 확보**이다.

따라서 다음과 같이 해석한다.

- **`K = 8`** → 존재 확인 및 약한 분포 관찰 가능
- **`K = 15`** → 후속 sense 비교를 시작할 만한 기본 evidence 확보
- **`K = 20+`** → 더 안정적이지만 운영 비용 증가

결론적으로, 본 설계에서는 다음을 채택한다.

```text
기본 중단조건: K >= 15
```

즉, 누적 slang evidence 15개를 확보하면 현재 단계의 목적에는 충분하다고 본다.

---

## 7. 최종 샘플링 프로세스

### Step 1. 초기 샘플링

```text
초기 샘플 수 n0 = 60
```

---

### Step 2. 1차 판정

초기 60개 문맥에서 slang hit 수를 `K60`라고 하자.

- `K60 = 0` → **drop**
- `K60 >= 1` → **추가 탐색 진행**

근거:

- floor 5%를 가정할 때 60개에서 0-hit 확률은 약 4.6%이므로,  
  `K60 = 0`이면 최소선에도 못 미칠 가능성이 높다.
- 반대로 1개 이상 잡히면 추가 evidence를 모을 가치가 있다.

---

### Step 3. 추가 탐색

추가 탐색은 batch 단위로 진행한다.  
구현 편의상 **20개씩 추가 샘플링**하는 방식을 기본값으로 둔다.

```text
batch size = 20
```

추가 탐색은 다음 둘 중 하나를 만족할 때까지 계속한다.

1. **누적 slang hit `K >= 15` 도달**
2. **누적 샘플 수 `n`이 해당 bucket의 `n_max` 도달**

---

## 8. `match_count` bucket별 최대 탐색량

초기 60개 이후 추가 탐색 상한은 `match_count` 구간에 따라 다르게 둔다.

| match_count 범위 | 단어 수 | 최대 탐색량 `n_max` |
|------------------|--------:|--------------------:|
| < 10K            | 667     | 100 |
| 10K ~ 50K        | 2,614   | 120 |
| 50K ~ 200K       | 1,534   | 140 |
| 200K ~ 1M        | 1,056   | 160 |
| > 1M             | 552     | 200 |

### 해석

- 작은 bucket은 문맥 풀이 상대적으로 작으므로 상한도 낮게 둔다.
- 큰 bucket은 evidence를 더 많이 확보할 가치가 있으므로 상한을 높게 둔다.
- 단, 초기 탐지 자체는 모든 단어에 동일하게 60개를 본다.

---

## 9. 최종 상태 정의

각 단어는 다음 셋 중 하나의 상태로 분류된다.

### 9.1 Drop
다음 조건을 만족할 때:

- 초기 60개에서 `K60 = 0`

의미:

- 보수적 floor 5% 기준에서도 slang evidence가 관찰되지 않음
- 현재 단계에서 추가 비용을 들일 가치가 낮음

### 9.2 Enough evidence
다음 조건을 만족할 때:

- 누적 slang hit `K >= 15`

의미:

- 후속 sense 분포 분석을 진행할 만한 기본 evidence 확보 완료

### 9.3 Holdout
다음 조건을 만족할 때:

- `K < 15`인데 `n = n_max` 도달

의미:

- slang evidence는 일부 있으나, 현재 비용 제한 안에서는 충분한 분포까지는 도달하지 못함
- 후속 우선순위 조정 또는 예외 검토 대상

---

## 10. 한 줄 요약

> 모든 단어를 먼저 60개 문맥으로 샘플링하고,  
> 60개에서 slang hit가 0개면 drop한다.  
> 1개 이상이면 20개씩 추가 탐색을 진행하여,  
> 누적 slang evidence가 15개에 도달하거나 `match_count` bucket별 최대 탐색량에 도달할 때까지 본다.

---

## 11. 의사코드

```python
def get_n_max(match_count: int) -> int:
    if match_count < 10_000:
        return 100
    elif match_count < 50_000:
        return 120
    elif match_count < 200_000:
        return 140
    elif match_count < 1_000_000:
        return 160
    else:
        return 200


def sample_until_threshold(match_count: int, initial_contexts, next_batch_provider):
    n = 60
    K = count_slang_hits(initial_contexts[:60])

    if K == 0:
        return {
            "decision": "drop",
            "sampled": 60,
            "slang_hits": 0,
        }

    n_max = get_n_max(match_count)

    while n < n_max and K < 15:
        batch = next_batch_provider(size=20)
        batch_hits = count_slang_hits(batch)
        K += batch_hits
        n += len(batch)

    if K >= 15:
        return {
            "decision": "enough_evidence",
            "sampled": n,
            "slang_hits": K,
        }
    else:
        return {
            "decision": "holdout",
            "sampled": n,
            "slang_hits": K,
        }
```

---

## 12. 주의사항 / 한계

1. **5% floor는 보수적 운영 가정이지, 실제 최소 slang 비율의 참값을 뜻하지 않는다.**
2. **`K >= 15`는 대표 sense 최종 확정 기준이 아니라, 후속 분포 분석을 위한 기본 evidence 확보 기준이다.**
3. **매우 약한 실제 slang은 상한에 도달해도 `K < 15`일 수 있으며**, 이런 경우는 `holdout`으로 남긴다.
4. 향후 summary 결과가 더 누적되면 다음 값들은 재보정할 수 있다.
   - floor ratio = 5%
   - 초기 샘플 수 = 60
   - 목표 slang hit 수 = 15
   - bucket별 `n_max`
